# `.specs/` — feature specifications & plans

Each feature lives in **one self-contained folder** `.specs/<name>/`. The `.specs/` root
holds only these feature folders — **no loose `*.md` files**.

## Per-feature layout

```
.specs/<name>/
  spec.md              # the short source spec you write by hand (the input)
  expanded/            # detailed design docs generated by /mp-spec-expander
    overview.md  data-model.md  architecture.md  ui.md  testing.md  open-questions.md  (+ feature-specific)
  plan/                # ordered task files generated by /plan-tasks
    00-overview.md     #   the plan manual — never implemented or moved
    NN-<slug>.md       #   one file per task, dependency-ordered
    implemented/       #   completed task files are moved here
  resolution-log.md    # running record of deviations, issues→resolutions, feedback
```

## Lifecycle

Create the folder and its `spec.md` yourself, then run the pipeline. Each stage stamps a
`status:` in `spec.md`'s YAML frontmatter:

| Stage | Command | Reads | Writes | `status:` |
|-------|---------|-------|--------|-----------|
| 1. Draft | *(you)* | — | `.specs/<name>/spec.md` | `draft` |
| 2. Expand | `/mp-spec-expander <name>` | `spec.md` | `expanded/` | `expanded` |
| 3. Plan | `/plan-tasks <name>` | `expanded/` | `plan/` + `resolution-log.md` | `planned` |
| 4. Ship | `/ship-plan <name>` | `plan/` | code + moves tasks to `plan/implemented/` | `shipped` |

Step 4 drives the `plan-implementer` agent task-by-task; each agent run appends to
`resolution-log.md` and moves its finished task file into `plan/implemented/`.

## Starting a new feature

```
mkdir .specs/<name>
# write .specs/<name>/spec.md  (a few paragraphs: problem, goals, rough approach)
/mp-spec-expander <name>
/plan-tasks <name>
/ship-plan <name>
```
